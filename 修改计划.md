你现在需要在仓库 `https://github.com/Thinking-builder/Flint.git` 中持续迭代开发，目标是把 Flint 从当前的线性任务执行器升级为一个更强的 VS Code 本地优先任务编排插件：支持任务并发执行，并显著增强“自我迭代推进任务执行”的能力。

当前我认为项目的核心问题是：任务执行模型过于线性，类似 `plan → execute → judge → revise once → finish`；Judge-then-revise 很快结束，缺少持续推进、失败反思、分支尝试、进度度量、任务图调度和长期记忆。你需要围绕这个方向持续改进代码，而不是只写文档或只做表层重构。

你的总体目标：

1. 阅读并理解 Flint 当前仓库结构、核心模块和执行流程。
2. 找出当前 orchestrator、scheduler、judge、provider、storage、workspace-tools 等模块中限制“并发执行”和“自我迭代推进”的设计问题。
3. 在不破坏现有 MVP 可运行性的前提下，逐步实现更强的 agentic task orchestration 架构。
4. 每一轮都必须遵循：分析 → 设计 → 小步实现 → 测试 → 自评 → 继续迭代。
5. 最终交付一组高质量代码修改、必要测试、清晰说明文档，以及后续路线建议。
6. 不要直接提交 PR，不要直接 push 到远程仓库。完成后给出 diff 总结和验证结果，等待我审阅。

重点改造方向：

一、改造自我迭代机制

当前不要简单地把 judge 重试次数从 1 改成 5。你需要设计一个更合理的迭代策略，例如：

* IterationPolicy
* ProgressLedger
* canonical verdict
* stagnation detection
* evidence-based verification
* retry / branch / rollback / ask-human 决策
* failed attempt reflection
* next hypothesis generation

目标是让任务不是 judge 失败后简单 revise 一次，而是能够根据失败原因、测试结果、进度变化和剩余预算持续推进。

建议实现的数据结构包括但不限于：

```ts
interface IterationPolicy {
  maxAttempts: number;
  maxToolIterationsPerStep: number;
  maxTotalToolCalls: number;
  maxRuntimeMs?: number;
  maxCostUsd?: number;
  stagnationWindow: number;
  minProgressDelta: number;
  allowBranching: boolean;
  requireHumanReviewOnRisk: boolean;
}

interface ProgressLedger {
  score: number;
  scoreHistory: number[];
  passedChecks: CheckResult[];
  failedChecks: CheckResult[];
  fixedFindings: Finding[];
  openFindings: Finding[];
  blockers: Blocker[];
  currentHypotheses: Hypothesis[];
  stagnationCount: number;
}
```

你可以根据实际代码结构调整命名和类型，但必须体现“进度度量”和“持续推进”的设计。

二、重构 verdict 逻辑

检查当前 judge / orchestrator 中是否存在 `verdict`、`overallVerdict`、deterministic checks、criteria checks 之间不一致的问题。

你需要统一最终判定逻辑，例如：

```ts
finalVerdict = mergeVerdicts({
  llmJudgeVerdict,
  deterministicCheckVerdict,
  criteriaVerdict,
  humanReviewVerdict
});
```

最终任务状态必须只能由 canonical final verdict 推导，避免出现 judge 原始 verdict 是 pass，但 deterministic checks 已经 fail，任务仍被标记为 completed 的情况。

三、增强 scheduler 的并发能力

当前 scheduler 很可能只是基于 `maxConcurrentTasks` 和空闲 agent 进行简单任务级调度。你需要评估并逐步改造为更接近 resource-aware scheduler 的设计。

目标包括：

* 支持任务级并发。
* 为未来子任务级并发留下接口。
* 引入 resource lock 概念。
* 避免多个任务同时写同一文件或同一目录。
* 支持 read/write lock 区分。
* 为 git worktree / temp workspace attempt 隔离预留结构。
* 保持默认行为简单可靠，不要一次性引入过复杂的分布式调度。

建议接口：

```ts
interface ResourceLock {
  type: "file" | "directory" | "workspace" | "terminal" | "git-worktree";
  path?: string;
  mode: "read" | "write";
}
```

四、把线性 plan step 逐步升级为 task graph 思路

不要一次性大改到不可运行。优先做兼容式演进。

当前如果是线性 `PlanStep[]`，可以先增加 TaskGraphNode 类型，并提供从旧 PlanStep 到新 TaskGraph 的转换层。

目标结构：

```ts
interface TaskGraphNode {
  id: string;
  title: string;
  objective: string;
  dependsOn: string[];
  status: NodeStatus;
  requiredEvidence: string[];
  allowedTools: string[];
  resourceLocks: ResourceLock[];
  verification: VerificationSpec[];
}
```

短期可以仍然按拓扑顺序执行，长期支持 DAG 并发。

五、加入 Reflection / Memory 机制

Flint 需要从失败中学习。你需要设计并实现一个轻量 ReflectionStore，用于保存任务执行经验：

```ts
interface Reflection {
  id: string;
  taskType: string;
  projectFingerprint: string;
  failureModes: string[];
  successfulStrategy?: string;
  commandsDiscovered: string[];
  filesTouched: string[];
  avoidNextTime: string[];
  createdAt: number;
}
```

每次任务完成、失败、暂停时，应该生成一条 reflection。后续相似任务开始时，应检索相关 reflection 并注入 planning context。

第一版可以使用本地 JSON / SQLite / existing storage abstraction 实现，不要引入过重依赖。

六、增强 verification-first 工作流

Flint 不应该只依赖 LLM judge。你需要优先利用确定性信号：

* npm test
* npm run build
* npm run lint
* TypeScript typecheck
* extension compile
* unit tests
* changed-file targeted tests
* diff summary
* static checks

请先自动识别项目 package scripts，再决定运行哪些命令。不要盲目运行危险命令。对耗时命令设置合理超时。

七、增加 attempt / branch 设计，但可以分阶段实现

理想方向是每个复杂任务可以创建多个 attempt：

* attempt_001：最小修改
* attempt_002：重构修改
* attempt_003：兼容层修改

每个 attempt 独立执行、验证、评分，最后选择最佳 diff。

第一阶段可以先实现数据结构和记录机制，不一定马上实现完整 git worktree。请设计时预留：

```ts
interface Attempt {
  id: string;
  taskId: string;
  strategy: string;
  status: AttemptStatus;
  startedAt: number;
  finishedAt?: number;
  patchSummary?: string;
  verificationResults: CheckResult[];
  score?: number;
}
```

八、保留人类审阅点

高风险修改必须进入 review_required，而不是自动继续。例如：

* 删除大量文件
* 修改 package lock
* 修改配置文件
* 修改安全相关逻辑
* 执行危险 shell 命令
* 测试无法确认但 judge 想判定 pass
* 多次失败后仍想继续扩大修改范围

你必须持续运行测试和类型检查。每一轮修改后都要尝试运行：
